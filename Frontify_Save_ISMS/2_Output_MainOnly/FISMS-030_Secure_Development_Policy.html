2 Introduction

2.1 Purpose

The purpose of this policy is to ensure information security is designed and implemented within the development lifecycle.

﻿

﻿

2.2 Scope

All employees and third-party users.

﻿

System development of the Frontify app.

﻿

﻿

2.3 Principle

Secure software and system engineering principles and standards are implemented and tested.

﻿

Information security and privacy are by design and default.

3 Secure Development Policy

3.1 Secure Development Coding Guidelines

Every code change must pass the agreed criteria before shipping to avoid human errors. Combined with the bug bounty program and unit tests covering some security aspects, it must be ensured to meet the best possible code security.

﻿

Software is designed and developed based on industry secure coding guidelines for the coding technology and the Open Web Application Security Project (OWASP).

﻿

The most common pitfalls and further information shall be adhered to by Engineers and documented in the Product Technology Wiki. The VP Technology or their designee is responsible for keeping this documentation up to date.

﻿

Additionally, regular know-how exchanges shall be implemented to keep the team up to date regarding potential changes.

﻿

Each new developer shall receive a mandatory code security slot during their onboarding process or, respectively, before accessing the development environment.

﻿

﻿

3.2 Segregation of Environments

There must be a clear segregation between development, testing, staging, and production environments to maintain systems' integrity, availability, and security. Engineers are required to adhere to the best practices outlined in the deployment process accessible in the Product Technology Wiki.

﻿

﻿

3.3 Development Code Repositories

Development code must be stored in a secure code repository (e.g., GitHub) that enforces and meets the requirements of the access control policy and segregation of duty.

﻿

Development code repositories must enforce version control and appropriate version archiving.

﻿

﻿

3.4 Development Code Reviews

The Code must be reviewed prior to release by skilled personnel other than the code author/developer.

﻿

The Code must be reviewed against the secure development coding guidelines.

﻿

The Code reviews must employ manual and automated techniques (e.g., Unit- and E2E-Tests).

﻿

﻿

3.5 Development Code Approval

Code must be approved before being pushed into test or production.

﻿

Artifacts (e.g., source code, libraries, software components, data) released under an open-source license must be requested, reviewed, tested, approved, and inventoried by the architecture team prior to utilization, following the procedures listed in the Product Technology Wiki.

﻿

﻿

3.6 Testing

All pre-production testing shall occur in a test environment.

﻿

The test environment shall mirror the production environment as closely as possible.

﻿

Application security testing shall be performed using manual and automated techniques.

﻿

Testing shall be performed as a minimum test for the OWASP top 10.

﻿

All public-facing web applications must be tested using manual or automated vulnerability security tools or methods at least monthly or after a significant change.

﻿

﻿

3.6.1 Existing Modules and Products

For changes (including bug fixes) on already active modules and products, no change management process or penetration test is needed.

﻿

Already active modules and products must automatically be part of the bug bounty program scope.

﻿

﻿

3.6.2 New Modules and Products

Prior to the initial release, an external penetration is needed for newly developed modules or products.

﻿

All penetration testing is conducted by an external specialist company.

﻿

All vulnerabilities identified in the newly developed modules or products as part of the testing phase are analyzed and/or mitigated (if feasible) before pushing to production or managed via risk management.

﻿

﻿

3.7 Test Data

Production data must never be used for testing or development.

﻿

Personal data must never be used for testing or development.

﻿

If sensitive information is required as part of the testing process, it must be approved by the Information Security Office.

﻿

﻿

3.8 Pushing Code to Production

Code must be pushed to production by approved personnel and is subject to the documented change control process.

﻿

The production environment is backed up before pushing code to production to facilitate rollback for a failed change.

﻿

Test data shall be removed before the application is pushed to production.

﻿

No development files or test data should be stored in the production environment.

4 Policy Compliance

4.1 Compliance Measurement

The Information Security Office will verify compliance to this policy through various methods, including but not limited to, business tool reports, internal and external audits, and feedback to the policy owner.

﻿

﻿

4.2 Exceptions

Any exception to the policy must be approved and recorded by the Information Security Office in advance and reported to the CISO.

﻿

﻿

4.3 Non-Compliance

An employee found to have violated the Frontify Information Security Management System may be subject to disciplinary action, up to and including termination of employment.

﻿

﻿

4.4 Continual Improvement

The policy is updated and reviewed as part of the continual improvement process.
